🚀 Platform Overview

Welcome to the Template Editor Platform documentation. This system provides a powerful, flexible way to create and manage website templates with a visual editor that allows real-time content editing without touching code.

Key Features

  • Visual Editing - Edit content directly in the browser with instant preview
  • Data Attribute System - Simple, declarative way to mark editable content
  • Component-Based - Reusable React components with variants
  • Type-Safe - Full TypeScript support throughout
  • Flexible Architecture - Write components your way, editor adapts to you
  • Auto-Detection - Editor automatically discovers editable fields

💡 Philosophy

Developers write components naturally. The editor adapts through data attributes. No complex patterns, no boilerplate, just React components with special attributes.

🏗️ Architecture

System Overview

The platform consists of four main layers:

Template Components
↓
Component Registry
↓
DOM Scanner
↓
Visual Editor

1. Template Components Layer

React components with data attributes marking editable content. Developers have full creative freedom.

2. Component Registry Layer

Maps section types and IDs to their React components using lazy loading for optimal performance.

3. DOM Scanner Layer

Automatically scans rendered HTML for data-type attributes and builds editing schema.

4. Visual Editor Layer

Provides UI for editing content, managing sections, and previewing changes in real-time.

Technology Stack

Layer Technology Purpose
Frontend React + TypeScript Component rendering and UI
State Management Zustand Editor state and template selections
Styling Tailwind CSS Utility-first styling
Build Tool Vite Fast development and optimized builds
Backend Node.js API and data persistence

⚡ Quick Start

Create Your First Section (5 Minutes)

Step 1: Write Component

// templates/my-template/sections/hero/MyHero.tsx
export const MyHero = ({ title, description, photos }) => {
  const heroImage = photos?.find(p => p.id === "hero_image");
  
  return (
    <section className="py-20">
      <h1 data-type="text" data-name="title">
        {title || "Welcome"}
      </h1>
      <p data-type="textarea" data-name="description">
        {description || "Your description here"}
      </p>
      <img 
        src={heroImage?.url || "/default.jpg"}
        data-type="image" 
        data-name="hero_image"
      />
    </section>
  );
};

Step 2: Register Component

// client/src/shared/utils/component-registry.ts
registry["hero:1"] = {
  component: lazy(() => 
    import("@templates/sections/hero/MyHero")
      .then(m => ({ default: m.MyHero }))
  )
};

Step 3: Done!

The editor automatically detects all 3 editable fields and creates appropriate inputs.

✅ That's It!

No builders, no factories, no complex setup. Just React components with data attributes.

📊 Data Flow

How Data Moves Through the System

User Edits Content
↓
Editor Updates State
↓
State Syncs to Server
↓
Component Re-renders
↓
DOM Updates

Data Flow Steps

  1. Initial Load - Template data loaded from server
  2. DOM Scan - Editor scans for data-type attributes
  3. Schema Generation - Creates editing schema from detected fields
  4. User Interaction - User edits content in sidebar
  5. State Update - Zustand store updates with new values
  6. Component Update - React re-renders with new props
  7. Persistence - Changes saved to server

Real-Time Updates

The system uses React's reactive nature to provide instant feedback:

// User types in editor
onChange={(value) => {
  // Update store
  updateSectionContent(sectionId, fieldName, value);
  
  // Component automatically re-renders with new value
  // No manual DOM manipulation needed
}}

🏷️ Data Attributes System

The Core Mechanism

Data attributes are the foundation of the editor system. They tell the editor what content is editable and how to edit it.

Supported Attributes

Attribute Purpose Example
data-type Defines the field type data-type="text"
data-name Unique identifier for the field data-name="title"
data-title Label shown in editor UI data-title="Main Heading"
data-placeholder Placeholder text for empty fields data-placeholder="Enter text"

Data Types

1. Text (data-type="text")

For short text content like headings, buttons, labels.

<h1 data-type="text" data-name="heading">
  {heading || "Default Heading"}
</h1>

2. Textarea (data-type="textarea")

For longer text content like paragraphs, descriptions.

<p data-type="textarea" data-name="description">
  {description || "Default description"}
</p>

3. Link (data-type="link")

For URLs and hyperlinks.

<a href={ctaLink || "#"} data-type="link" data-name="cta_link">
  Click Here
</a>

4. Image (data-type="image")

For images and background images.

<img 
  src={heroImage?.url || "/default.jpg"}
  data-type="image" 
  data-name="hero_image"
/>

⚠️ Important

data-name must be unique within a section. Use descriptive names like hero_title instead of generic names like text1.

🔍 DOM Scanning

How It Works

The DOM scanner automatically discovers editable fields by:

  1. Rendering the component
  2. Querying for elements with data-type attribute
  3. Extracting metadata (name, title, placeholder)
  4. Determining field type
  5. Building editing schema
  6. Creating appropriate UI controls

Scanner Implementation

// Simplified scanner logic
const scanSection = (sectionElement) => {
  const fields = [];
  
  // Find all elements with data-type
  const editableElements = sectionElement.querySelectorAll('[data-type]');
  
  editableElements.forEach(el => {
    const type = el.getAttribute('data-type');
    const name = el.getAttribute('data-name');
    const title = el.getAttribute('data-title') || name;
    
    fields.push({
      type,
      name,
      title,
      value: getCurrentValue(el, type)
    });
  });
  
  return fields;
};

💡 Automatic Detection

You never need to manually configure fields. The scanner finds them automatically by reading your component's HTML.

🎭 Sections & Variants

Section Concept

A section is a reusable piece of a page (hero, features, products, etc.).

Variants

Multiple design variations of the same section type. Users can switch between variants without losing content.

Creating Variants

// Multiple variants for hero section
export const HeroSimple = ({ title }) => (
  <section className="text-center">
    <h1 data-type="text" data-name="title">{title}</h1>
  </section>
);

export const HeroWithImage = ({ title, photos }) => {
  const heroImage = photos?.find(p => p.id === "hero_image");
  return (
    <section className="grid grid-cols-2">
      <h1 data-type="text" data-name="title">{title}</h1>
      <img src={heroImage?.url} data-type="image" data-name="hero_image" />
    </section>
  );
};

export const HeroFullWidth = ({ title, photos }) => {
  const bgImage = photos?.find(p => p.id === "background");
  return (
    <section 
      style={{ backgroundImage: `url(${bgImage?.url})` }}
      data-type="image" 
      data-name="background"
    >
      <h1 data-type="text" data-name="title">{title}</h1>
    </section>

};

💡 Content Preservation

When switching variants, content is preserved if field names match. For example, if both variants have data-name="title", the title content will be maintained.

📋 Best Practices

Naming Conventions

Component Names

  • Use PascalCase: HeroSection, ProductGrid
  • Be descriptive: HeroWithBackgroundImage not Hero2
  • Include variant info: HeroSimple, HeroCentered

Data Names

  • Use snake_case: hero_title, cta_link
  • Be specific: hero_title not title
  • Use prefixes for groups: feature_1_title, feature_1_desc
  • Avoid generic names: main_heading not text1

Section IDs

  • Format: page-section-variant
  • Example: home-hero-1, products-grid-2
  • Be unique across the template

⚠️ Critical

data-name must be unique within each section. Duplicate names will cause editing conflicts.

⚡ Performance

Optimization Strategies

1. Lazy Loading

All components are lazy-loaded by default:

registry["hero:1"] = {
  component: lazy(() => import("./Hero"))
};

2. Image Optimization

  • Use appropriate image formats (WebP, AVIF)
  • Implement lazy loading for images
  • Provide responsive images with srcset
  • Compress images before upload

3. Memoization

Use React.memo for expensive components:

export const ProductCard = React.memo(({ product }) => {
  // Component logic
});

4. Debouncing

Debounce auto-save operations:

const debouncedSave = useMemo(
  () => debounce(saveToServer, 500),
  []
);

5. Code Splitting

Each section is automatically code-split, loading only when needed.

✅ Built-in Optimizations

The platform handles lazy loading, code splitting, and component registry automatically. You just write components.
