Overview

This server provides a comprehensive API for managing multi-tenant e-commerce stores with template-based page building capabilities. The server supports both local development and production deployments with Cloudflare R2 storage integration.

Key Features:
  • Multi-tenant Architecture: Supports multiple stores using subdomain-based routing
  • Template System: Flexible template management with page-level customization
  • Real-time Style Generation: Dynamic CSS generation based on store settings
  • Cloud Storage: Cloudflare R2 integration with local fallback
  • Development Tools: Mock data and hot reloading for development
  • Publishing Workflow: Draft and published states for content management

Architecture Overview

The server follows a clean architecture pattern with clear separation of concerns:

API Layer

  • RESTful endpoints for all operations
  • Express.js routing with middleware
  • Request validation and error handling
  • Multi-tenant request routing

Business Logic

  • Store management operations
  • Template processing
  • Style generation pipeline
  • File upload handling

Data Layer

  • SQLite database with migrations
  • JSON-based store configuration
  • Template metadata storage
  • Published content management

Storage Layer

  • Cloudflare R2 cloud storage
  • Local file system fallback
  • CSS file generation and caching
  • Asset management and organization

Getting Started

To start using the API:

  1. Set up your environment: Configure database and storage settings
  2. Choose your store: Access via subdomain (e.g., demo.localhost)
  3. Configure settings: Use store endpoints to set up colors, fonts, and layout
  4. Select templates: Apply page templates for different sections
  5. Publish changes: Make your store live with the publish endpoint

Database Schema

The database uses SQLite with a simple yet powerful schema designed for multi-tenant store management. The schema supports automatic migrations and includes built-in optimization for JSON data storage.

Database Design Principles:
  • JSON Storage: Store configurations are stored as JSON for flexibility
  • Multi-tenancy: Subdomain-based store isolation
  • Publishing Workflow: Separate draft and published states
  • Template System: Metadata-driven template management

Tables

stores

Primary table for storing store data and configurations. This table uses a hybrid approach combining structured columns with flexible JSON storage.

Usage Notes:
  • The json column contains draft/editable store data
  • The published_json column contains live/published data
  • Subdomain must be unique across all stores
  • Store ID is a UUID for global uniqueness
Column Type Description Example
store_id TEXT (PRIMARY KEY) Unique identifier for the store (UUID) "123e4567-e89b-12d3-a456-426614174000"
template_id TEXT Template identifier used by the store "retail-v1"
json TEXT Store configuration data (JSON) - draft state "{\"storeSettings\": {...}}"
published_json TEXT Published version of store data (JSON) - live state "{\"storeSettings\": {...}}"
subdomain TEXT (UNIQUE) Subdomain for multi-tenant routing "demo"

templates

Table for storing available templates. This table acts as a registry for all template types and their metadata.

Template Structure:
  • Templates are organized by category (e.g., ecommerce, restaurant)
  • Each template has a corresponding folder in the filesystem
  • Preview images help users visualize templates
  • Templates can be versioned through the ID field
Column Type Description Example
id TEXT (PRIMARY KEY) Template unique identifier "retail-v1"
category TEXT Template category for grouping "ecommerce"
folder TEXT Template folder name in filesystem "retail-v1"
title TEXT Template display title for UI "Retail v1"
description TEXT Template description for users "A modern, responsive template for retail stores."
preview_image TEXT Preview image URL or path "/templates/retail-v1/preview.jpg"

Database Migrations

The server automatically handles database schema migrations on startup:

Migration 1: Base Schema

  • Creates the stores table
  • Sets up basic store structure
  • Establishes primary keys and constraints

Migration 2: Published State

  • Adds published_json column
  • Enables draft/publish workflow
  • Handles existing data migration

Migration 3: Multi-tenancy

  • Adds subdomain column
  • Enables subdomain-based routing
  • Sets up unique constraints

Migration 4: Templates

  • Creates templates table
  • Seeds initial template data
  • Sets up template registry

API Endpoints

The API follows RESTful conventions and provides comprehensive endpoints for store management, template handling, and content publishing. All endpoints are scoped to the current store based on the subdomain.

API Usage Guidelines:
  • Store Context: All endpoints automatically detect the store from the request's Host header
  • Error Handling: Consistent error format with appropriate HTTP status codes
  • JSON Responses: All responses use JSON format with data wrapper
  • CORS Support: Configured for cross-origin requests in development

Store Management

These endpoints manage store configuration data, including settings, content, and custom data. They operate on the draft version of store data.

GET /api/v1/store/:key

Retrieve a specific key value from the store's draft data. This endpoint is used to fetch configuration settings, page content, or any custom data stored under a specific key.

Use Cases:
  • Loading store settings (colors, fonts, layout)
  • Retrieving page content and components
  • Fetching custom configuration data
  • Reading draft changes before publishing
Parameters:
  • key (path): The data key to retrieve (e.g., "storeSettings", "pages", "products")
Response: 
{
  "data": {
    "colors": {
      "primary": "#6366f1",
      "secondary": "#8b5cf6"
    },
    "fonts": {
      "heading": "Inter",
      "body": "Inter"
    }
  } || null
}

POST /api/v1/store/:key

Store a value for a specific key in the store's draft data. This endpoint automatically merges new data with existing data and triggers style generation for settings changes.

Special Behavior:
  • When key is "storeSettings", automatically generates CSS styles
  • Data is merged with existing store data (not replaced)
  • Auto-creates store if it doesn't exist
  • Updates styleUrl when settings are modified
Parameters:
  • key (path): The data key to store
  • value (body): The value to store (any JSON-serializable data)
Request Example: 
{
  "value": {
    "colors": {
      "primary": "#6366f1",
      "secondary": "#8b5cf6"
    }
  }
}
Response: 
{
  "success": true
}

GET /api/v1/published/store/:key

Retrieve a specific key value from the store's published data. This endpoint returns the live/public version of store data, not the draft version.

Important:
  • Returns null if no published data exists
  • Use this for public-facing applications
  • Separate from draft data to prevent showing unfinished changes
Parameters:
  • key (path): The data key to retrieve
Response: 
{
  "data": "value or null"
}

Multi-Tenant Architecture

Store Resolution

The server uses subdomain-based multi-tenancy:

  • demo.localhost → demo store
  • store.example.com → store store
  • localhost → demo store (default)

Store Data Structure

Each store contains:

  • storeSettings: Theme colors, fonts, and configuration
  • pageTemplates: Array of page template selections
  • styleUrl: Generated CSS URL
  • Custom data keys as needed

Storage Configuration

Cloudflare R2 Integration

The server supports Cloudflare R2 for file storage:

Environment Variables:
  • R2_ACCESS_KEY_ID: R2 access key
  • R2_SECRET_ACCESS_KEY: R2 secret key
  • R2_ENDPOINT: R2 endpoint URL
  • R2_BUCKET_NAME: R2 bucket name
  • R2_PUBLIC_URL: Public URL for R2 files

Fallback Storage

If R2 is not configured, files are stored locally in client/public/uploads/.

Style Generation

The server generates custom CSS based on store settings:

Store Settings Structure

{
  "colors": {
    "primary": "#4272FF",
    "secondary": "#ACBA12",
    "text": "#1D293D"
  },
  "fonts": {
    "heading": "sans-serif",
    "body": "sans-serif"
  }

Generated CSS includes:

  • CSS custom properties for colors and fonts
  • Base styles and typography
  • Tailwind CSS utilities

Error Handling

All endpoints return consistent error responses:

{
  "error": "Error description"
}
Common error codes:
  • 400: Bad Request (missing parameters, invalid data)
  • 404: Not Found (store not found, data not found)
  • 500: Internal Server Error (database errors, parsing errors)

Development vs Production

Development Mode

  • Uses Vite for hot module reloading
  • Loads mock data from template files
  • Supports live template editing

Production Mode

  • Uses published data only
  • Static file serving
  • Optimized for performance

Security Considerations

  • File upload size limit: 10MB
  • Subdomain validation and filtering
  • JSON parsing with error handling
  • R2 configuration validation

Database Migrations

The server handles automatic migrations:

  1. published_json column addition
  2. subdomain column addition
  3. Templates table creation
  4. Seed data population
